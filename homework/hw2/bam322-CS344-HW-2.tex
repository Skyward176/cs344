\begin{problem}
  You are given an array \( A[1..n] \) with the following properties:
  \begin{itemize}
    \item \( A[1] > A[2] \) (the first element is greater than the second).
    \item \( A[n-1] \leq A[n] \) (the second-to-last element is less than or equal to the last).
  \end{itemize}

  A \textbf{local minimum} is an element \( A[x] \) such that:
  \[
    A[x-1] \geq A[x] \leq A[x+1].
  \]
  For example, in the array \( A = [9, 7, 7, 2, 1, 3, 7, 5, 4, 7, 3, 3, 4, 8, 6, 9, 6] \), there are six local minima.

  Design an \( O(\log n) \)-time algorithm to find a local minimum.

\end{problem}
\begin{solution}
  \

  \textbf{Algorithm:}
  \
  \begin{enumerate}
    \item First, set low to index 1 and high to index n.
    \item Set middle to be the floor of (low + high) / 2.
    \item If A[middle] is a local minimum of its neighbors, return it.
    \item If A[middle] is not a local minimum, then check if $A[middle-1] < A[middle]$. If this is the case, then there must be a local minimum in the left half of the array. Otherwise, there must be a local minimum in the right half of the array. Run recursively on the corresponding half.
  \end{enumerate}
  \textbf{Proof:}
  \
  We are given an array where the first and last elements form the left and right boundaries of local minima, respectively. We check the middle in between each iteration. If it isn't a local minimum, we split the array into halves, and check again. This results in a maximum of logn iterations. Because we always follow the half where there is a value smaller than the middle (and the extremes are necessarily larger), we always move towards a minimum.
  %=================================================
  % 3. RUN TIME ANALYSIS %=====================================================

  \textbf{Runtime:}
  \

  Because we split the array in half each iteration, this runs in O(logn) time. It follows the same recursion as binary search, since it mechanically does the same.
\end{solution}

\begin{problem}
  Suppose you are given two sorted arrays $A[1 . . n]$ and $B[1 . . n]$ containing distinct integers. Describe a $O(\log n)$ algorithm to find the median (meaning the $n$th smallest element) of the union $A \cup B$. For example, given the input

  $$
  A[1 . .8]=[0,1,6,9,12,13,18,20] \quad B[1 . .8]=[2,4,5,8,17,19,21,23]
  $$

  your algorithm should return the integer 9. [Hint: What can you learn by comparing one element of $A$ with one element of $B$ ?]

\end{problem}

\begin{solution}
  \

  \textbf{Algorithm:}
  \
  \begin{enumerate}
    \item Compare the medians of both arrays.
    \item \textbf{Rule 1}: In the case of whichever array has a smaller median, all elements below it are in the n/2 smallest numbers (smaller than the overall median of the array).

      All elments above the median of the array with the larger median are larger than the overall median array (in the n/2 largest numbers).

      Therefore, we make a subarray of array 1 and a subarray of array 2, removing the n/2 smallest elements from the array with the smallest median and the same number from the top of the array with the larger median.
    \item Find the median of the subarrays and compare them. Again, rule 1 applies.
    \item When our we are left with one item in each subarray, return the smaller one.
    \item This gives the n-th smallest number in both arrays, or their median.
  \end{enumerate}

  \textbf{Proof:}
  \

  In the base case of two arrays of length 1, the median (when defined as the nth smallest) is the smaller of the two numbers.

  When we throw away the upper half of the array with the larger median and the lower half of the array with the smaller median, we are halving the list of potential overall medians. We know this is true because in the case of the smaller median array, all values less than its median are also smaller than the median of the larger array, which means they are in the smaller n/2 items. By the same logic, in the larger array, those above the median are in the larger n/2 items.

  We now have halved our search space and found the first half of the first n smallest numbers. now we take the medians again, similarly restricting down our upper and lower bounds for the median again. Eventually, this always goes to the base case where we pick the smaller and receive the nth smallest element.
  \textbf{Runtime:}
  \

  Because each partition cuts the two arrays in half, we are running in O(logn) time.
\end{solution}

\begin{problem}
  Write an algorithm in pseudocode:

  \begin{itemize}
    \item Input: An array $A$ with $n$ distinct (non-equal) elements
    \item Output-1: numbers $x$ and $y$ in $A$ that minimize $|x-y|$, where $|x-y|$ denotes absolute-value(x-y). (If there are multiple closest pairs, you only have to return one of them.)
    \item Output-2: a pair of numbers $w$ and $z$ in $A$ that minimize $|w+z|.$
  \end{itemize}
  The run-time should be significantly better than $O(n^2)$.

\end{problem}

\begin{solution}
  \

  \textbf{Algorithm:}
  \

  \begin{verbatim}
    sortedArray = mergesort(A)
    minDistance = (infinity, (-1,-1))
    for int i=0 to (n-2):
      difference = absoluteValue(sorted_arr[i+1] - sorted_arr[i])
      if difference<minDistance[0]:
        minDistance = (difference, (i+1, i))

    i = 0
    j = n-1
    minSum = (infinity, (-1,-1))
    while i < j:
      sum = absoluteValue(sortedArray[i] + sortedArray[j])
      if sum<minSum[0]:
        minSum = (sum, (i,j))
      if absoluteValue(sortedArray[i]) > absoluteValue(sortedArray[j]):
        i++
      else:
        j--

    output_1 = (sortedArray[minDistance[1][0]], sortedArray[minDistance[1][1]])
    output_2 = (sortedArray[minSum[1][0]], sortedArray[minSum[1][1]])
  \end{verbatim}

  \textbf{Proof:}
  \
  This works because in a sorted(increasing) array, the next element is necessarily the nearest neghbor in the increasing direction. Therefore, once sorted, we can do n comparisons to see which two neighbors have the smallest absolute value difference between them. This gives us output 1.

  \textbf{Runtime:}
  \

  We first run mergesort, which takes nlogn time. Then we iterate through every item in the array twice, which takes 2N time. This simplifies to nlogn time, with the sort being the only significant factor.
\end{solution}

\begin{problem}
  An inversion (swap) in an array $A[1 . . n]$ is a pair of indices $(i, j)$ such that $i<j$ and $A[i]>A[j]$. The number of inversions in an $n$-element array is between 0 (if the array is sorted) and $\binom{n}{2}$ (if the array is sorted backward). Describe and analyze an algorithm to count the number of inversions in an $n$-element array in $O(n \log n)$ time. [Hint: Modify mergesort.]
\end{problem}

\begin{solution}
  \

  \textbf{Algorithm:}
  \

  For brevity- we can view this as just running mergesort with this modification to the merge function:

  \begin{verbatim}
    merge(list1, list2, inversions):
      i = 0 # index counter for list 1
      j = 0 # index counter for list 2

      while i<length(list1) and j<length(list2)
        if list1[i] <= list2[j]:
          output.append(list1[i])
          i++
        else:
          output.append(list2[j])
          j++
          inversions += length(list1) - i

      while i < length(list1):
        output.append(list1[i])
        i++

      while j < length(list2):
          output.append(list2[j])
          j++

      return output, inversions

  \end{verbatim}
  \

  \textbf{Proof:}
  \

  In mergesort, we split the array into n subarrays. Then, we merge them back together into order. Whenever we find that an item in the left half is bigger than one on the right, we perform swaps. Each time we do this, we add to our number of inversions the number of values that are exclusively larger in the left half than the value in the right half, in other words, the number of inversions resolved.
  When we have sorted the whole array and propagated our total, we have managed to count the number of inversions resolved by the algorithm.
  \

  \textbf{Runtime:}
  \

  The division step of mergesort runs log n times. We run n merges. This gives O(nlogn) time complexity.
\end{solution}

\begin{problem}
  Given an array \( A \) of length \( n \), and two indices \( i \) and \( j \) with \( i \leq j \), define \( \text{Prod}(A,i,j) \) as the product of elements from \( A[i] \) to \( A[j] \), i.e., \( A[i] \times A[i+1] \times \cdots \times A[j] \). If \( i = j \), then \( \text{Prod}(i,j) = A[i] \).

  For example, if \( A = [3, 0.1, 5, 20, 4] \), then \( \text{Prod}(A,1,3) = 0.1 \times 5 \times 20 = 10 \).

  Now, consider the following problem MaxProd(A):

  \begin{itemize}
    \item \textbf{Input}: An array \( A \) of length \( n \) where all elements are positive, but may include fractional values (e.g., \( A[i] = 0.1 \) is valid).
    \item \textbf{Output}: The maximum possible value of \( \text{Prod}(A, i, j) \) for some subarray \( A[i..j] \).
  \end{itemize}

  For example, if \( A = [3, 0.2, 4, 0.5, 1, 4, 0.3, 2] \), the maximum product is \( 8 \), which is achieved by the subarray \( [4, 0.5, 1, 4] \).

\begin{enumerate}[label=\alph*)]
  \item

    Write pseudocode for a recursive algorithm \( \text{MaxProd}(A) \) that solves the problem in \( O(n\log n) \) time. You only need to provide the pseudocode and the recurrence relation for the algorithm.

    \textit{Hint}: The recurrence formula is similar to one we've used before, so you don't need to prove that it solves to \( T(n) = O(n \log n) \).

    \textit{Note}: While there are faster, non-recursive algorithms for this problem, you must use a recursive algorithm for this homework.

    \begin{solution}

      I will skip straight to the O(N) alg.

    \end{solution}

  \item Now, write pseudocode for a recursive algorithm that solves the problem in \( O(n) \) time. As in Part 1, provide the pseudocode and the recurrence relation for the algorithm.

    \textit{Hint}: Similar to the Max Profit problem covered in class, solve a helper problem \( \text{MaxProdX}(i, j) \), which gives additional information beyond just the maximum product.

    If you're confident with the \( O(n) \) algorithm, you can directly write a single solution for both Part 1 and Part 2. However, if you're unsure, first implement the \( O(n \log n) \) solution for Part 2, then write the separate \( O(n) \) algorithm.
\end{enumerate}
\end{problem}

\begin{solution}
\

\textbf{Algorithm:}
\

  \begin{verbatim}
    def maxProdRecursion(array, index):
      if index == 0:
        return (array[0], array[0])

      (bestProduct, prevBestProduct) = maxProdRecursion(A, index - 1)

      newProduct = max(A[index], (prevBestProduct*A[index]))

      if newProduct > bestProduct:
        bestProduct = newProduct

      return(bestProduct, newProduct)
    def MaxProd(A):
      lastIndex = len(A) - 1
      bestProduct, _ = maxProdRecursion(A, lastIndex)
      return(bestProduct)
  \end{verbatim}
\
\textbf{Proof:}

\
In the base case, this just gives back the value of the first item in the array.
For the recursion, our function returns the best product found in the whole array so far, and the best product that ends at the previous index.
For our current index, we are iterating on, the best product we can finding ending in that index is either current element by itself, or it's the current element multiplied by the best product of the subarray
ending right before it. We check both options and save the winner as our new best product ending at this index. This works because we aren't dealing with negatives.
Finally, we compare the product of this new subarray to the overall best product we've seen anywhere in the array so far.
If it's bigger, it becomes the new overall best. Since we do this for every single element as the recursion unwinds,
we are guaranteed to have found the absolute largest contiguous product by the time we return to the main wrapper function.

\textbf{Runtime:}

\
This algorithm is O(N) because each recursive call makes only one other recursive call (constant work per layer) and it goes from an array of length n to an array of length 1, or N recursive calls.

This satisfies the recursion $T(N) = T(N-1) + O(1)$ where the constant work is just the comparisons for tracking the current best subarray product we've found.
\end{solution}

\begin{problem}
You are given two strings $A$ and $B$ of length $n$, each consisting of possibly repeated characters from an alphabet of size $M$. Two strings $A$ and $B$ are said to be \textbf{anagrams} of one another if it is possible to reorder the characters in $A$ to obtain string $B$ (without removing or adding any characters). For instance, the strings ``ELEVEN PLUS TWO'' and ``TWELVE PLUS ONE'' are anagrams of one another (think of empty-space  also as a character in the alphabet).

\smallskip

Your goal is to design an algorithm that decides if a given pair of strings are anagrams of one another.
You may assume that all characters in the alphabet can represented by distinct integers in the set $\set{1,2,\ldots,M}$, and that you can convert any character to its integer representation in $O(1)$ time (note that $M$ can be much larger than $n$).
\smallskip

\begin{enumerate}
  \item[(a)] Design an algorithm for this problem with worst-case  runtime of $O(n+M)$.

    \begin{solution}
      \

      \textbf{Algorithm:}
      \

      let counts be an array of length M, initialize it with zeros

      for every character in string A, increment the corresponding index in the counts array

      for every character in string B, decrement the corresponding index in the counts array

      if any value in the counts array is nonzero, return false. Else, it is true.

      \
      \textbf{Proof:}

      \
      This is trivially true due to our given definition of anagram. They must have the same exact number of the same exact characters to be anagrams of each other.

      Therefore, by first creating a count of which characters are present in one string and seeing if the second string "cancels out" the counts, we determine if the second string has the exact same characters in the same number as the first.
      \

      \textbf{Runtime:}

      The initialization and checking of the count array each take M time. Iterating over each string takes n time. Therefore we have O(2n + 2M), which we simplify to O(n+M)
    \end{solution}

  \item[(b)] Design a randomized algorithm for this problem with worst-case expected runtime of $O(n)$.

    \begin{solution}
      \

      \textbf{Algorithm:}
      \

      Create hash table

      for every character in string A, if it is not in the hash table, add it to the hash table with a value of one.
      If it is in the hash table, increment the value by one

      for every character in string B, if it is present in the hash table and the value is greater than zero, decrement the value by one.
      Otherwise, if it is not in the hash table or the value stored is zero, return false.

      If we don't fail before this point, we return true.

      \
      \textbf{Proof:}

      \
      The same as before, this is trivially true due to our given definition of anagram. They must have the same exact number of the same exact characters to be anagrams of each other.

      In this case, we know this works because first- the strings are the exact same length. It is impossible for the two to have the exact same number of certain characters but one of them contain a character unique from the other.
      Because they are the same length, if we never tried to set a value below zero in the hash table, we know they had identical number of each character contained in them.
      \

      \textbf{Runtime:}
      This runs in O(n) time because we iterate over each string once O(n+n) which simplifies to just O(n). If we use a randomized hash function and universal hashing, we get constant time lookups, so we can ignore access time as well, giving us the O(n) algorithm.
    \end{solution}

\end{enumerate}

\end{problem}

\begin{problem}
Suppose we have an array $A[1:n]$ which consists of numbers $\set{1,\ldots,n}$ written in some arbitrary order (this means that $A$ is a \emph{permutation} of the set $\set{1,\ldots,n}$).
Our goal in this problem is to design a very fast randomized algorithm that can find an index $i$ in this array such that $A[i] \mod 5 = 0$, i.e., $A[i]$ is divisible by five.
For simplicity, in the following, we assume that $n$ itself is a multiple of $5$ and
is at least $5$ (so a correct answer always exist).

For instance, if $n=10$ and the array is $A=[8,7,2,5,4,6,3,1,10,9]$, we can output either of indices $4$ or $9$.

\begin{enumerate}
  \item[(a)]  Suppose we sample an index $i$ from $\set{1,\ldots,n}$ uniformly at random. What is the probability that $i$ is a correct answer, i.e., $A[i] \mod 5 = 0$?

    \begin{solution}
      $p = \frac{n/5}{n}$
    \end{solution}

  \item[(b)] Suppose we sample $m$ indices from $\set{1,\ldots,n}$ uniformly at random and with repetition. What is the probability that none of these indices is a correct answer?  \grade{5}
\end{enumerate}
\begin{solution}
  $p = (\frac{4n/5}{n})^M$
\end{solution}

Now, consider the following simple algorithm for this problem:

\textbf{Find-Index-1$(A[1:n])$:}
\begin{itemize}
  \item Let $i=1$.  While $A[i] \mod 5 \neq 0$, sample $i \in \set{1,\ldots,n}$ uniformly at random. Output $i$.
\end{itemize}

The proof of correctness of this algorithm is straightforward and we skip it in this question.

\begin{enumerate}
  \item[(c)] What is the worst-case \textbf{expected} running time of \textbf{Find-Index-1$(A[1:n])$}? Remember to prove your answer formally.
\end{enumerate}

\begin{solution}
  The worst case expected runtime will be O(1). This is because our worst case expectation of our random sample is that it will take five tries to find a value that is a multiple of five.
\end{solution}

The problem with \textbf{Find-Index-1} is that in the worst-case (and not in expectation), it may actually never terminate! For this reason, let us consider
a simple modification to  this algorithm as follows.

\textbf{Find-Index-2$(A[1:n])$:}
\begin{itemize}
  \item For $j=1$ to $n$:
    \begin{itemize}
      \item Sample $i \in \set{1,\ldots,n}$ uniformly at random and if $A[i] \mod 5 = 0$, output $i$ and terminate; otherwise, continue.
    \end{itemize}
  \item If the for-loop never terminated, go over the array $A$ one element at a time to find an index $i$ with $A[i] \mod 5 =0$ and output it as the answer.
\end{itemize}

Again, we skip the proof of correctness of this algorithm.

\begin{enumerate}
  \item[(d)] What is the \textbf{worst-case running time} of \textbf{Find-Index-2$(A[1:n])$}? What about its worst-case \textbf{expected} running time? Remember to prove your answer formally.

\end{enumerate}

\end{problem}

\begin{solution}
The worst case is O(n), where it had to run a linear search after managing to not sample a multiple of five by terrible luck. The worst case-expected runtime is still O(1) because we still expect to draw a multiple of five every five tries.
\end{solution}
